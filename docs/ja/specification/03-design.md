# 3. 設計

## 3.1 ハードリンクによる保全

一致するログファイルが現れると（inotify の create / close-write / moved-to イベント、または定期的な full resync 経由）、`pod-log-preserver` は保全ディレクトリ配下にハードリンクを作成し、元ファイルの相対パス（`<ns>_<pod>_<uid>/<container>/`）を再現する。

- **アクティブログ**（`<n>.log`）はリンク先の名前にタイムスタンプ接尾辞を付加してリンクされるため、同一のアクティブファイルが複数回ローテートしても衝突しない。
- **ローテート／圧縮済みログ**（`<n>.log.<ts>`、`<n>.log.<ts>.gz`）は元の名前を保持する。
- 重複作業は 2 つの方法で回避される。inode チェック（同一 inode を指すハードリンクが宛先ディレクトリに既に存在するか）と、宛先名の衝突チェックである。

ハードリンクは inode を共有するため、kubelet が元ファイルを削除した後も、保全ファイルはログのバイト列を生かし続ける。

## 3.2 Tail DB 確認によるクリーンアップ

クリーンアップパスは固定間隔で実行される。各保全ファイルについてリンクカウントを stat する。

- `Nlink > 1` —— 元ファイルはまだ存在する。スキップする。
- `Nlink == 1` —— **孤児（orphan）**。元ファイルは削除済みで、保全されたハードリンクだけが残っている。削除候補となる。

孤児となったローテートログについて、`pod-log-preserver` はログエージェントの tail DB（サイクルごとに一度だけロードされ、読み取り専用）を参照する。ファイルが **収集済みと確認される（confirmed consumed）** のは、以下の場合である。

1. 少なくとも 1 つの tail DB に、そのファイルの inode に対応する行があり、記録された `name` が `/<relPath>` で終わっていること（先頭のセパレータによってパス境界での一致を保証し、無関係なファイルとの inode 番号の衝突を防ぐ）。**かつ**
2. そのような一致行を持つすべての DB において、記録されたオフセットがファイルサイズに達していること。

確認済みのファイルは即座に削除され、エージェントのファイルディスクリプタが解放される。

**複数の tail input:** そのファイルに対応する行を持たない DB があっても削除はブロックされない。それは「未完了」ではなく「まだ発見されていない」とみなされる。これが安全なのは、ファイルが孤児になるのはハードリンクされてから複数回のローテーションサイクルを経た後であり、その時点までに保全ツリーを正当に tail しているエージェント（サブセカンドの refresh interval を持つ）はすでにそのファイルを発見しているはずだからである。結果として、新しい tail input はここで進捗が尊重されるために同じ DB グロブを使用しなければならない。

## 3.3 Age ベースのフォールバック

DB を利用したクリーンアップが無効（グロブが空）な場合、DB が読み取れない場合、またはファイルが確認済みでない場合、削除はファイルの mtime に基づく age しきい値にフォールバックする。

- `.gz` でない孤児は `CLEANUP_MAX_AGE_MIN` を使用する。
- `.gz` の孤児は `CLEANUP_GZ_MAX_AGE_MIN` を使用する（圧縮済みログはより長く保持される）。

保全ツリー配下の空ディレクトリは、各サイクルの終わりに機会的に削除される。

## 3.4 Namespace フィルタ

オプションのカンマ区切りの `NAMESPACE_FILTER` は、グロブパターン（例: `team-*`）を用いてパスの最初のセグメント（`<ns>_<pod>_<uid>` の `<ns>`）と照合し、保全対象を一致する namespace に限定する。空の場合はすべての namespace が対象となる。
