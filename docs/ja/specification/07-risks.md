# 7. リスクと状況

## 7.1 リスク

| # | リスク | 対策 |
|---|------|-----------|
| 1 | 保全ディレクトリと watch ディレクトリが異なるファイルシステム上にあり、ハードリンクが不可能になる | 起動時のハードリンクテストが、いかなる作業の開始前にも早期に失敗する（§4.1）。 |
| 2 | エージェントが読み終える前に保全ファイルが削除される | 削除は確認優先（confirmation-first）である。孤児は tail DB でオフセット ≥ サイズが示された場合にのみ削除され、そうでなければ age しきい値を待つ（§3.2–3.3）。 |
| 3 | inode 番号の衝突により tail DB 上で無関係なファイルと一致してしまう | 記録された `name` も `/<relPath>` で終わっていることを要求し、パス境界での一致に固定する（§3.2）。 |
| 4 | inotify のキューオーバーフローによりイベントが失われる | キューオーバーフローイベントは full resync を発生させ、定期的な resync も無条件に実行される（§5.1）。 |
| 5 | 新しい tail input の初回発見が遅く、削除と競合する | ファイルが孤児になるのはハードリンクされてから複数回のローテーションサイクルを経た後であり、その時点までにサブセカンドの refresh を持つエージェントは発見済みである。新しい input は DB グロブを共有しなければならない（§3.2）。 |
| 6 | root と hostPath での実行は広い権限を伴う | このタスクには必須であり、Kubernetes API へのアクセスは要求せず、tail DB も読み取り専用である（§4.3）。 |

## 7.2 検証済みの前提

- 社内の前身システムは本番環境で fluent-bit の tail DB に対して稼働しており、ハードリンクによる保全 + DB 確認によるクリーンアップという挙動はそこで実証済みである。
- **本リリースにおいて再検証すべき事項:** pure-Go の `modernc.org/sqlite` ドライバが、実際の fluent-bit の WAL tail DB を読み取り専用で読み、以前の CGO ドライバと同一の結果を返すこと —— 実装中に、コピーした実際の DB に対して検証済み。
- **自動再検証:** ハイブリッド e2e ハーネス（`test/e2e/`）が、出荷イメージと実際の fluent-bit WAL tail DB に対して、保全 → fluent-bit による確認 → クリーンアップの全ループを CI 実行のたびに検証すること —— 上記の一度きりの手動確認を継続的なゲートに置き換える。

## 7.3 未解決の論点

- 未確認のログを決して破棄したくない運用者向けに、「確認済みのみ、age による削除は行わない」厳格モードを提供するかどうか。
- fluent-bit 以外のログエージェント（異なる tail DB スキーマを持つ）をプラガブルなリーダーの背後でサポートするかどうか。
