# 1. 概要

## 1.1 背景

EKS Auto Mode では kubelet がマネージドであり、そのログローテーション設定 —— `containerLogMaxSize`（デフォルト 10MB）と `containerLogMaxFiles`（デフォルト 5）—— はカスタマイズできない。そのため、ログエージェントが収集する速度よりも速くログを出力するコンテナでは、ローテートされたログファイルが一度も読まれないまま kubelet によって削除され、そのログ行が永久に失われる可能性がある。

## 1.2 ゴール

- ログエージェントが確実に収集したと確認できるまで、ローテートされたすべての Pod ログを保全する。
- ノードに測定可能な負荷を追加しない。ハードリンク（コピーではない）、inotify イベントループ、ログエージェントの tail DB に対する単一コネクションの読み取り専用クエリのみを用いる。
- ディスクを速やかに解放する。収集が確認できた時点で即座に保全ファイルを削除し、必要以上に保持しない。
- SQLite のみを外部依存とする自己完結型 DaemonSet として、distroless の static イメージ上で提供する。

## 1.3 非ゴール

- ログシッパーではない。`pod-log-preserver` はログの内容を読み取ったり解析したり転送したりすることは一切なく、ファイルを保全・解放するだけである。収集はあくまでログエージェントの役割である。
- 汎用のバックアップツールではない。保全は収集の確認または age（経過時間）しきい値によって境界づけられ、無期限に保持されるわけではない。
- ノード間の集約や外部データストアは持たない。

## 1.4 用語

- **アクティブログ（Active log）**: kubelet が現在書き込み中の `<n>.log` ファイル。
- **ローテートログ（Rotated log）**: kubelet がローテートした `<n>.log.<timestamp>`（`.gz` の場合もある）。
- **保全ファイル（Preserved file）**: 保全ディレクトリ配下に作成される、Pod ログと同一 inode を指すハードリンク。
- **孤児（Orphan）**: リンクカウントが 1 に落ちた保全ファイル。元の Pod ログは削除されており、保全されたハードリンクだけが残っている状態。
- **Tail DB**: ログエージェント（fluent-bit）が inode ごとに記録する SQLite データベースで、読み取ったバイトオフセットと認識しているパスを保持する。
- **確認済み（Confirmed）**: 保全ファイルについて、tail DB に記録されたオフセットがファイルサイズに達していることを示している状態 —— エージェントがそのファイルを最後まで読み終えたことを意味する。

## 1.5 ログ収集パイプラインにおける位置づけ

`pod-log-preserver` はログエージェントのパスの中には存在せず、その隣に位置する。kubelet は `/var/log/pods` 配下にログを書き込み、エージェントはそれを tail する。`pod-log-preserver` はそれらをハードリンクで退避させ、エージェント自身の進捗 DB を読み取り専用で参照することで、エージェントがそのファイルを処理し終えた時点でのみ、保全済みのコピーを削除する。
